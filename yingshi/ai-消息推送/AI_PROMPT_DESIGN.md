# AI文章分析和评分功能 - Prompt设计文档

## 模型配置

- **模型ID**: `glm-z1-flash`
- **模型名称**: `glm-z1-flash`
- **配置格式**: `model_id,model_name`

## Prompt设计

### 角色定位
```
你是一位专业的技术文章分析专家。
```

### 输入内容
- 文章标题
- 文章内容（限制8000字符，避免超出token限制）

### 输出格式

必须输出纯JSON格式，不要包含任何markdown代码块标记。

```json
{
    "one_sentence_summary": "一句话总结，50字以内",
    "summary": "详细摘要，200-300字",
    "main_points": [
        "主要内容点1，50字以内",
        "主要内容点2，50字以内",
        "主要内容点3，50字以内",
        "主要内容点4，50字以内"
    ],
    "key_quotes": [
        "文章金句1，原文引用，30-50字",
        "文章金句2，原文引用，30-50字",
        "文章金句3，原文引用，30-50字"
    ],
    "ai_score": 85,
    "tags": ["标签1", "标签2", "标签3", "标签4"],
    "related_readings": [
        {"title": "相关阅读标题1", "description": "为什么相关"},
        {"title": "相关阅读标题2", "description": "为什么相关"}
    ]
}
```

### 字段详细要求

#### 1. one_sentence_summary（一句话总结）
- **字数**: 50字以内
- **要求**: 
  - 必须包含文章的核心主题和主要价值点
  - 精炼、准确、有信息量
  - 能让读者快速理解文章的核心内容

#### 2. summary（详细摘要）
- **字数**: 200-300字
- **要求**:
  - 系统性地概括文章的主要观点、技术细节、解决方案和结论
  - 确保读者不需要阅读原文就能理解文章的核心内容
  - 结构清晰，逻辑连贯
  - 包含文章的背景、主要内容、技术要点和结论

#### 3. main_points（主要内容）
- **数量**: 3-5个核心内容点
- **字数**: 每个点50字以内
- **要求**:
  - 每个点应该是一个独立且重要的观点或技术要点
  - 按照文章中的重要程度排序
  - 涵盖文章的核心技术、方法和观点

#### 4. key_quotes（文章金句）
- **数量**: 2-4个精彩句子
- **字数**: 每个句子30-50字
- **要求**:
  - 必须是原文的完整引用，不能修改
  - 应该是最有代表性、启发性或技术价值的句子
  - 选择最能体现文章核心观点或技术亮点的句子

#### 5. ai_score（AI评分）
- **范围**: 0-100分，整数
- **评分标准**:
  - **文章质量**（0-30分）: 内容深度、逻辑性、可读性
  - **技术价值**（0-30分）: 创新性、实用性、参考价值
  - **信息完整性**（0-20分）: 是否覆盖主题、是否有实际案例
  - **写作水平**（0-20分）: 结构清晰度、表达准确性
  
- **分数等级**:
  - **90-100分**: 优秀，深度技术文章，有重要价值
  - **80-89分**: 良好，高质量技术文章，值得阅读
  - **70-79分**: 中等，有一定价值，但可能不够深入
  - **60-69分**: 一般，基础内容，参考价值有限
  - **0-59分**: 较差，内容质量不高

#### 6. tags（标签）
- **数量**: 3-5个关键词标签
- **要求**:
  - 应该反映文章的主要技术领域、主题和关键词
  - 使用通用的技术术语
  - 涵盖文章的技术栈、主题和关键概念

#### 7. related_readings（相关阅读）
- **数量**: 2-3个相关阅读建议（可选）
- **要求**:
  - 每个建议包含标题和简要描述
  - 描述应说明为什么这个阅读与文章相关
  - 如果没有合适的相关阅读，可以不填（空数组）

### 输出要求

1. **格式要求**:
   - 必须输出有效的JSON格式
   - 不要包含任何markdown代码块标记（如```json或```）
   - 所有字符串必须使用双引号
   - 确保JSON格式完全正确，可以被直接解析

2. **内容要求**:
   - 如果某个字段无法分析或不存在，使用空值（空字符串、空数组或null）
   - 所有分析内容必须基于文章实际内容
   - 不要添加文章中没有的信息

3. **质量要求**:
   - 分析要准确、客观、有价值
   - 评分要公正，基于文章实际质量
   - 摘要要全面，能帮助读者快速理解文章

## API调用格式

### Dify API格式

**端点**: `POST {ai_api_base_url}/chat-messages`

**Headers**:
```
Authorization: Bearer {ai_api_key}
Content-Type: application/json
```

**Payload**:
```json
{
    "inputs": {},
    "query": "{prompt内容}",
    "response_mode": "blocking",
    "conversation_id": "",
    "user": "story_{story_id}"
}
```

**响应**:
```json
{
    "answer": "{AI返回的分析结果JSON}",
    ...
}
```

## 配置示例

在`.env`文件或环境变量中配置：

```bash
# AI模型配置（格式：model_id,model_name）
AI_MODEL_CONFIG=glm-z1-flash,glm-z1-flash

# AI API基础URL（智谱OpenAI兼容接口）
AI_API_BASE_URL=https://open.bigmodel.cn/api/paas/v4

# AI API密钥
AI_API_KEY=your_zhipu_api_key
```

> 如果在自建的 Dify 等网关中承载 Prompt，需要将 `AI_API_BASE_URL` 指向对应服务，并使用该服务要求的模型 ID（通常是工作流 ID）。请避免在直连智谱接口时填写 Dify 工作流 ID，否则会返回 “模型不存在” 的 400 错误。

## 注意事项

1. **内容长度限制**: 文章内容限制为8000字符，避免超出token限制
2. **超时设置**: API请求超时设置为120秒，因为AI分析可能需要较长时间
3. **错误处理**: 如果API调用失败或返回格式错误，会记录日志并抛出异常
4. **缓存机制**: 已分析的文章会缓存结果，避免重复分析（可通过`force_refresh`参数强制刷新）
5. **自动分析**: 文章加载时如果没有分析结果，会自动触发AI分析（延迟2秒）
