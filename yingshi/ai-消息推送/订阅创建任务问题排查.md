# RSS订阅创建任务问题排查指南

## 问题现象

订阅创建任务一直处于"处理中"（pending 或 updating）状态，无法完成。

## 问题原因

根据代码分析，最常见的原因是：

1. **Scheduler 服务未运行**（90% 的问题都是这个原因）
   - Scheduler 服务负责从任务队列获取任务并分发给 Worker
   - 如果 Scheduler 未运行，任务会一直停留在队列中，不会被处理

2. **Worker 服务未运行**
   - Worker 服务负责执行实际的任务（如抓取订阅源）
   - 如果 Worker 未运行，Scheduler 无法调用 Worker 执行任务

3. **服务连接问题**
   - Scheduler 无法连接到 Harbor 服务（API 服务）
   - Scheduler 无法连接到 Worker 服务

4. **任务队列异常**
   - 任务过期未清理
   - 任务创建失败

## 解决方案

### 方案1：启动 Scheduler 服务（最常见）

如果 Scheduler 服务未运行，立即启动它：

```bash
 cd /path/to/rss-subscribe
source $(conda info --base)/etc/profile.d/conda.sh
conda activate rssant
bash rundev-scheduler.sh
```

### 方案2：使用统一脚本启动所有服务

推荐使用统一脚本启动所有服务：

```bash
 cd /path/to/rss-subscribe
bash start-all-services.sh start
```

这会启动：
- API 服务（端口 6788）
- Worker 服务（端口 6793）
- **Scheduler 服务（端口 6790）** ← 关键服务

### 方案3：检查并重启服务

```bash
# 查看服务状态
bash start-all-services.sh status

# 重启 Scheduler 服务
bash start-all-services.sh restart scheduler

# 查看 Scheduler 日志
bash start-all-services.sh logs scheduler
```

## 任务处理流程

理解任务处理流程有助于排查问题：

```
1. 用户创建订阅
   ↓
2. API 服务创建 FeedCreation（状态: PENDING）
   ↓
3. API 服务调用 batch_find_feed → 创建 WorkerTask 到数据库队列
   ↓
4. **Scheduler 服务从队列获取任务** ← 如果这里停止，任务不会处理
   ↓
5. Scheduler 调用 Worker 服务执行任务
   ↓
6. Worker 服务处理任务：
   - 更新 FeedCreation 状态为 UPDATING
   - 执行 find_feed 操作
   - 完成后调用 harbor_rss.save_feed_creation_result
   ↓
7. Harbor 服务更新 FeedCreation 状态为 READY 或 ERROR
   ↓
8. 前端刷新显示订阅
```

## 常见问题

### Q1: 为什么 Scheduler 服务这么重要？

**A:** Scheduler 服务是任务处理的核心调度器：
- 它从数据库任务队列（WorkerTask 表）中获取任务
- 它将任务分发给 Worker 服务执行
- 如果 Scheduler 未运行，任务会一直停留在队列中，永远不会被执行

### Q2: 如何查看任务队列状态？

```bash
python manage.py shell
>>> from rssant_api.models import WorkerTask
>>> WorkerTask.objects.count()  # 查看任务总数
>>> WorkerTask.stats()  # 查看任务统计
>>> from rssant_api.models import FeedCreation, FeedStatus
>>> FeedCreation.objects.filter(status=FeedStatus.PENDING).count()  # 查看待处理任务
```

### Q3: 任务长时间处于 UPDATING 状态怎么办？

如果任务超过4小时仍处于 UPDATING 状态，可能已经卡死。系统会自动重试：
- `task_service.py` 中的 `_fetch_find_feed_task` 方法会定期检查
- 超过4小时的 UPDATING 或 PENDING 任务会被重新加入队列

也可以手动检查：

```bash
python manage.py shell
>>> from rssant_api.models import FeedCreation, FeedStatus
>>> from django.utils import timezone
>>> from datetime import timedelta
>>> now = timezone.now()
>>> long_updating = FeedCreation.objects.filter(
...     status=FeedStatus.UPDATING,
...     dt_updated__lt=now - timedelta(hours=4)
... )
>>> long_updating.count()
```

### Q4: 如何查看服务日志？

```bash
# 查看 Scheduler 日志
screen -r rssant-scheduler

# 查看 Worker 日志
screen -r rssant-worker

# 查看 API 日志
screen -r rssant-api

# 或使用统一脚本
bash start-all-services.sh logs scheduler
bash start-all-services.sh logs worker
bash start-all-services.sh logs api
```

在 screen 中按 `Ctrl+A` 然后按 `D` 可以退出（服务继续运行）。

### Q5: 如何检查服务之间的连接？

检查服务配置：

```python
python manage.py shell
>>> from rssant_config import CONFIG
>>> print(f"Harbor URL: {CONFIG.harbor_url}")
>>> print(f"Worker URL: {CONFIG.worker_url}")
```

确保这些 URL 可以访问。

## 验证修复

修复后，验证服务是否正常工作：

1. **检查所有服务都在运行**
   ```bash
   bash quick_check.sh
   ```

2. **创建一个测试订阅**
   - 在前端创建一个新的订阅
   - 观察是否能在短时间内从"处理中"变为"已完成"

3. **查看任务队列是否被处理**
   ```bash
   python manage.py shell
   >>> from rssant_api.models import WorkerTask
   >>> WorkerTask.objects.count()  # 应该很快变为 0
   ```

## 预防措施

1. **使用统一启动脚本**
   - 使用 `start-all-services.sh` 确保所有服务都启动

2. **使用进程管理工具**
   - 使用 screen 或 tmux 管理服务
   - 或使用 systemd 等服务管理工具

3. **定期检查服务状态**
   - 定期运行 `quick_check.sh` 检查服务状态
   - 设置监控告警

4. **查看日志**
   - 定期查看服务日志，及时发现问题
